Smart Redirect for Erasure Coding

Smart Redirect is a HyperStore feature that improves write availability, read availability, and durability of erasure coded data.

A standard way of implementing erasure coding in a consistent hashing based storage ring such as HyperStore is to identify for each uploaded object a list of k+m endpoint nodes on which to write the object's k+m fragments, with the composition of the endpoint node list being determined by a hash of the object name and by the token range responsibilities of the nodes. If your configured consistency requirements call for successful writes of all k+m fragments, an object upload attempt will fail if any of the object's k+m endpoint nodes are unavailable. With weaker write consistency requirements, unavailable nodes among the endpoint list will result in fewer than k+m fragments being written, or -- if too many of the endpoint nodes are down -- the object upload request failing despite the relaxed consistency requirements.

With Smart Redirect, rather than identifying just k+m endpoint nodes on which to write the k+m fragments, HyperStore identifies k+m+l eligible endpoint nodes, where "l" is a number of additional nodes that can be used for the erasure coded fragment writes in case any of the other endpoint nodes are unavailable (because those nodes are down; or are in maintenance mode; or have been marked as temporarily down by the S3 Service). In this way the system increases the likelihood of successful writes of erasure coded objects. Under strong write consistency requirements, for a given object upload Smart Redirect makes it more likely that all k+m fragments can be written and that consequently the upload will succeed. Under weaker write consistency requirements, for a given object upload Smart Redirect makes it more likely that the write requirements will not only be met but exceeded, thereby improving future read availability and durability for that object.

For each storage policy that uses erasure coding, the system automatically determines an "l" value at the time that the policy is created and that "l" value remains in effect for the life of the storage policy. For example, if for a particular storage policy k+m+l = 4+2+2, then for each object uploaded to a bucket that uses that storage policy the system will identify 8 nodes as eligible for hosting one of the object's 6 fragments. As long as at least 6 of those 8 nodes are available, all 6 of the object's fragments can be written. Note that in no case will more than k+m fragments be written -- the extra "l" nodes are utilized for writes only if some of the other endpoints are unavailable.

For detailed information on S3 write and read availability under various combinations of cluster size and storage policy configuration, see Storage Policy Resilience to Downed Nodes.

How the System Determines an "l" Value: Single-DC Storage Policies

For a single-DC erasure coding storage policy the system automatically determines an "l" value at policy creation time based on the number of nodes in the data center above and beyond the storage policy's k+m value, but not to exceed m and not to exceed 3. (The capping of the "l" value at 3 is because the Smart Redirect feature entails some degree of additional overhead when erasure coded data is repaired or rebalanced.) Put formally, for a single-DC erasure coding storage policy the "l" value is determined by the following equation:

  • l = min(m, #nodes-(k+m), 3)

How the System Determines an "l" Value: Multi-DC Storage Policies

For a multi-DC replicated erasure coding storage policy, on object upload the system identifies k+m+l eligible endpoint nodes in each DC, and tries to write k+m object fragments in each DC.

For such storage policies the system automatically determines an "l" value at policy creation time, with the "l" value being constrained by the participating DC that has the fewest nodes above and beyond the k+m value. Formally, the "l" value for such a policy is determined by this pair of equations:

  • #nodes = min(#nodesInDC1, #nodesInDC2...)
  • l = min(m, #nodes-(k+m), 3).

For example, for 4+2 or 9+3 erasure coding storage policies replicated across two DCs:

# Nodes in DC1 # Nodes in DC2 "k"+"m" "l" Comment
6 6 4+2 0 Neither DC has nodes in excess of k+m, so l = 0.
7 6 4+2 0 The smaller DC (DC2) has no nodes in excess of k+m, so l = 0. The l of 0 is applicable to each DC (there cannot be different l values for different DCs).
8 7 4+2 1 The smaller DC (DC2) has only one node in excess of k+m, so l = 1. The l of 1 is applicable to each DC. So on object upload, in each DC the system will identify 7 (k+m+l) eligible endpoints on which to try to write 6 (k+m) fragments.

For a multi-DC distributed (non-replicated) erasure coding storage policy, on object upload the system identifies fragmentsPerDC+l eligible endpoint nodes in each DC, and tries to write fragmentsPerDC in each DC.

For such storage policies the system automatically determines an "l" value at policy creation time, with the "l" value being constrained by the participating DC that has the fewest nodes above and beyond the number of object fragments to be written per DC. Formally, the "l" value for such a policy is determined by the following equation:

  • l = min(m, #nodes - #fragmentsPerDC, 3)

For example, for 7+5 erasure coding storage policies distributed across 3 DCs:

10 6 5 7+5 4 1 The smallest DC (DC3) has only one node in excess of the Fragments Per DC value, so l = 1. The l of 1 is applicable to each DC. So on object upload, in each DC the system will identify 5 (fragmentsPerDC+l) eligible endpoints on which to try to write 4 (fragmentsPerDC) fragments.

The "l" value that the system assigns to an erasure coding storage policy when the policy is created cannot be changed. It remains in effect for the life of the storage policy.

The Impact of Adding Nodes to Your Cluster

A storage policy's "l" value is automatically determined by the system at the time of storage policy creation and cannot be changed afterward. Consequently, adding nodes to your cluster will not impact the "l" values associated with your existing erasure coding storage policies. However, adding nodes to your cluster may allow for larger "l" values when you create new erasure coding storage policies.
